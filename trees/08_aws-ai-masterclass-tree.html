Module 8. Essentials of Prompt Engineering ├─── A. Introduction to Prompt Engineering │ ├─── A1. What Prompt Engineering Is │ │ ├─── Prompt quality directly affects output quality — the module's central premise │ │ └─── Prompting bridges human intent and AI interpretation │ ├─── A2. The AnyCompany Scenario │ │ └─── A vague prompt from a finance company produced an unrelated smart-home analysis — the module's running example │ └─── A3. Why Weak Prompts Fail │ ├─── Lack of Specificity — The request doesn't narrow down what's actually wanted │ ├─── Missing Domain Context — The model has no signal about the industry or audience │ ├─── Unclear Output Structure — No indication of format, length, or organization │ └─── Generic Responses — The result: outputs that could apply to almost any topic ├─── B. Understanding Prompts │ ├─── B1. What a Prompt Is, and Its Benefits │ │ ├─── Enhances Model Performance — Better-guided models produce more accurate, on-target output │ │ ├─── Improves Output Safety — Reduces the chance of harmful or inappropriate content │ │ ├─── Adds Domain-Specific Behavior — Tailors generic model knowledge to a specific field or task │ │ └─── Produces Higher-Quality Responses — Clearer instructions consistently yield better-formed answers │ ├─── B2. The Four Core Prompt Components │ │ ├─── Instructions — Defines the task to perform, e.g. “Determine which orders can be fulfilled” │ │ ├─── Context — Background that tailors the output to domain or audience, e.g. “ecommerce inventory management task” │ │ ├─── Input Data — The actual content being processed, e.g. “orders and inventory quantities” │ │ └─── Output Indicator — Specifies the expected format, e.g. “fulfillment status” │ └─── B3. Negative Prompting │ └─── Negative prompting increases the safety and appropriateness of responses by restricting specific undesired behaviors ├─── C. Modifying Prompts and Inference Parameters │ ├─── C1. Prompt Refinement and Inference Parameters │ │ └─── Inference parameters and prompt wording are two separate levers on the same output │ ├─── C2. Temperature, Top P, and Top K │ │ ├─── Temperature — Controls randomness/creativity directly │ │ ├─── Temperature — Low (0.2): conservative, predictable, more coherent │ │ ├─── Temperature — High (1.0): creative, diverse, potentially less coherent │ │ ├─── Top P — Probability-based token filtering │ │ ├─── Top P — Low: smaller probability pool, more focused output │ │ ├─── Top P — High: wider probability pool, more diverse language │ │ ├─── Top K — Limits choices to the top K most probable words │ │ ├─── Top K — Low: more constrained responses │ │ └─── Top K — High: larger vocabulary selection, greater creativity │ ├─── C3. Length Parameters │ │ └─── Maximum Length caps token count; Stop Sequences end generation at a defined boundary │ └─── C4. AnyCompany's Prompt Improvements │ ├─── Added Finance Industry Context — Grounded the request in the correct domain │ ├─── Specified SMB Audience — Narrowed the intended reader │ ├─── Defined Report Structure — Removed ambiguity about output format │ ├─── Set Professional Tone — Constrained the register of the writing │ ├─── Increased Maximum Length — Allowed room for a fuller report │ └─── Increased Temperature and Top P — Traded some predictability for more creative framing, appropriate once structure was locked down ├─── D. Best Practices for Prompt Engineering │ └─── D1. Nine Core Best Practices │ ├─── Be Clear and Concise — Use natural language and direct instructions, not overly technical wording │ ├─── Be Clear and Concise — Good: “What is the sum of these numbers?” vs. bad: “Compute the sum total…” │ ├─── Include Context — Specify industry, audience, and business purpose to improve relevance │ ├─── Use Directives — State the output type clearly: summary, poem, analysis, table, JSON │ ├─── Specify Desired Output — Define length, tone, structure, and included/excluded content │ ├─── Start with Interrogatives — Frame requests with Who, What, Where, When, Why, How │ ├─── Provide Example Outputs — Demonstrate expected formatting or reasoning directly │ ├─── Break Up Complex Tasks — Divide into subtasks; ask the model to think step by step │ ├─── Experiment and Iterate — Compare prompt versions and refine based on output quality │ └─── Use Prompt Templates — Standardized, reusable structures improve consistency across workflows ├─── E. Prompt Engineering Techniques │ ├─── E1. Zero-Shot Prompting │ │ └─── Zero-shot: no examples, relies on pretrained knowledge, fast but less reliable for complex tasks │ ├─── E2. Few-Shot Prompting │ │ └─── Few-shot: examples included, improves consistency and formatting accuracy over zero-shot │ └─── E3. Chain-of-Thought (CoT) Prompting │ ├─── Mathematics — Multi-step calculations benefit from explicit intermediate steps │ ├─── Financial Analysis — Sequential reasoning through assumptions and figures │ ├─── Logic Problems — Step-by-step elimination of possibilities │ └─── Multi-Step Reasoning — Any task where the answer depends on an ordered chain of sub-conclusions ├─── F. Prompt Misuses and Risks │ ├─── F1. Prompt Misuse and Adversarial Prompting │ │ └─── Improperly designed or malicious prompts can compromise security, privacy, fairness, and system integrity │ └─── F2. Main Categories of Prompt Risk │ ├─── Poisoning — Corrupts training data, not the prompt interface itself │ ├─── Hijacking and Prompt Injection — Manipulates model behavior at inference time via crafted prompts │ ├─── Exposure — Unintentional disclosure of sensitive or confidential information │ ├─── Prompt Leaking — Disclosure of hidden system prompts or internal instructions │ └─── Jailbreaking — Circumvents safety restrictions to force prohibited output ├─── G. Poisoning Attacks │ ├─── G1. What Poisoning Is │ │ ├─── Poisoning's goal: corrupt outputs, introduce bias, cause unsafe responses, reduce trustworthiness │ │ └─── Poisoning happens at training/fine-tuning time, not at inference time │ ├─── G2. Examples and Real-World Implications │ │ ├─── Biased Training Examples — Injecting racist or biased examples into training data │ │ ├─── Manipulated Financial Data — Distorting predictions by manipulating financial datasets │ │ ├─── Malicious Code Samples — Inserting malicious code into coding training datasets │ │ └─── Misinformation in Corpora — Feeding misinformation into knowledge corpora used for training │ └─── G3. Mitigation Strategies │ ├─── Dataset Validation — Systematically checking data before it enters training │ ├─── Human Review Processes — Manual review catches what automated checks miss │ ├─── Bias Detection — Statistical checks for skew across training examples │ ├─── Data Provenance Tracking — Knowing exactly where each training example came from │ ├─── Secure Fine-Tuning Pipelines — Controlling who can introduce new data into a fine-tuning run │ └─── Regular Auditing — Ongoing checks, not a one-time validation pass ├─── H. Hijacking and Prompt Injection │ ├─── H1. What Prompt Injection and Hijacking Are │ │ ├─── Prompt injection targets the runtime prompt, not training data (contrast with Poisoning) │ │ └─── Goals: generate misinformation, bypass restrictions, produce malicious instructions, leak hidden instructions │ ├─── H2. Common Prompt Injection Techniques │ │ ├─── “Ignore previous instructions” — Directly attempts to override the system prompt │ │ ├─── “Act as an unrestricted AI” — Attempts to reframe the model's own identity and constraints │ │ ├─── “Pretend this is fictional” — Uses fictional framing to justify otherwise-restricted content │ │ └─── “For educational purposes only” — Uses a stated legitimate purpose to lower the model's guard │ ├─── H3. Poisoning vs. Prompt Injection │ │ └─── Poisoning = training data, long-term. Prompt Injection = runtime prompt, immediate. │ └─── H4. Mitigation Strategies │ ├─── Input Sanitization — Cleaning user input before it reaches the model │ ├─── Instruction Hierarchy Enforcement — Ensuring system instructions can't be overridden by user text │ ├─── Output Filtering — Catching problematic output after generation, before it reaches the user │ ├─── Human Oversight — Manual review for high-risk interactions │ ├─── Context Isolation — Preventing user input from bleeding into system-level context │ └─── Retrieval Safeguards — Protecting RAG pipelines from injected instructions hidden in retrieved documents ├─── I. Exposure and Prompt Leaking │ ├─── I1. Exposure │ │ ├─── Personal Information — Names, addresses, and other direct identifiers │ │ ├─── Financial Records — Account details, transaction history │ │ ├─── Medical Records — Protected health information │ │ ├─── Browsing History — Behavioral data revealing personal habits │ │ └─── Purchase History — Transaction records tied to individual customers │ ├─── I2. Prompt Leaking │ │ ├─── “Tell me your instructions” — Directly asks the model to reveal its system prompt │ │ ├─── “Repeat your hidden prompt” — Attempts to extract the exact system-level text │ │ └─── “What were you told before this conversation?” — Probes for prior context or configuration │ ├─── I3. Exposure vs. Prompt Leaking │ │ └─── Exposure = sensitive data disclosure. Prompt Leaking = system instruction disclosure. │ └─── I4. Mitigation Strategies │ ├─── Data Anonymization — Removing identifying details before data is usable by the model │ ├─── Access Control — Limiting who and what can query sensitive systems │ ├─── Encryption — Protecting data both at rest and in transit │ ├─── Prompt Isolation — Keeping system prompts separated from user-accessible context │ ├─── Retrieval Restrictions — Limiting what a RAG pipeline can surface to a query │ └─── Redaction Systems — Automatically stripping sensitive content from output before delivery ├─── J. Jailbreaking │ ├─── J1. What Jailbreaking Is │ │ ├─── Roleplay — Asking the model to adopt a persona without the usual constraints │ │ ├─── Hypotheticals — Framing prohibited content as a fictional or theoretical scenario │ │ ├─── Emotional Manipulation — Applying social pressure to get the model to comply │ │ ├─── Multi-Step Prompts — Building toward a prohibited output gradually across several turns │ │ └─── Instruction Overrides — Directly attempting to cancel prior safety instructions │ ├─── J2. Common Jailbreaking Approaches │ │ ├─── Character Simulation — “Pretend you are a hacker” or “act as a fictional villain” │ │ ├─── Recursive Instruction Attacks — Repeatedly re-framing the request to work around a refusal │ │ ├─── Prompt Chaining — Building the prohibited request across multiple linked prompts │ │ └─── Obfuscation Techniques — Disguising the actual request through encoding or indirection │ ├─── J3. Mitigation Strategies │ │ ├─── RLHF — Reinforcement Learning from Human Feedback aligns model behavior with safety expectations │ │ ├─── Safety Fine-Tuning — Additional training specifically targeting refusal behavior │ │ ├─── Output Moderation — Screening generated content before it reaches the user │ │ ├─── Adversarial Testing — Deliberately probing for jailbreak vulnerabilities before attackers do │ │ ├─── Behavioral Monitoring — Watching for patterns of repeated jailbreak attempts │ │ └─── Human Review Systems — Escalating suspicious interactions for manual review │ └─── J4. Prompt Injection vs. Jailbreaking │ └─── Prompt Injection = manipulate a specific output. Jailbreaking = remove safety restrictions broadly. ├─── K. Defending Against Prompt Risks │ ├─── K1. Four Categories of Defense │ │ ├─── Organizational Best Practices — Establish AI governance policies, audit prompts/outputs regularly, restrict access, monitor abnormal behavior, train employees │ │ ├─── Technical Defenses — Prompt filtering, output moderation, context segmentation, access control, retrieval isolation, secure logging │ │ ├─── Model-Level Defenses — RLHF alignment, adversarial training, safety fine-tuning, content moderation layers │ │ └─── Human-Centered Defenses — Human-in-the-loop review, ethical oversight boards, escalation mechanisms, incident response plans │ ├─── K2. Industry-Specific Defense Examples │ │ ├─── Healthcare — Mask patient identifiers │ │ ├─── Finance — Restrict transaction prompt access │ │ ├─── HR — Audit recruitment AI for bias │ │ ├─── Government — Enforce classified data filtering │ │ └─── Ecommerce — Prevent prompt-based fraud generation │ └─── K3. Common Indicators of Prompt Abuse │ ├─── Repeated Override Attempts — Multiple tries to cancel or bypass system instructions │ ├─── Requests for Hidden Prompts — Directly asking to see system-level instructions │ ├─── Suspicious Roleplay Scenarios — Framing that exists mainly to lower safety guardrails │ ├─── Malware or Illegal Action Requests — Direct or indirect requests for clearly prohibited content │ └─── Attempts to Reveal Confidential Data — Probing for information the user shouldn't have access to └─── L. Practical Applications and Exam-Level Insights ├─── L1. Why These Risks Matter in Real Organizations │ └─── A single prompt exploit can cascade into data leaks, manipulated outputs, customer harm, and compliance violations ├─── L2. The Mental Model: Attack Type to Target │ ├─── Poisoning — Targets Training Data │ ├─── Prompt Injection — Targets Prompt Instructions │ ├─── Exposure — Targets Sensitive Information │ ├─── Prompt Leaking — Targets System Prompts │ └─── Jailbreaking — Targets Safety Constraints └─── L3. Common Exam Traps └─── Three recurring exam traps: Leaking vs. Exposure, Injection vs. Jailbreaking, Poisoning vs. inference-time attacks

Developing Generative AI Solutions
Module 5. Developing Generative AI Solutions ├─── A. The Generative AI Application Lifecycle │ ├─── A1. Lifecycle Phases │ │ ├─── Six phases: Define → Select FM → Improve → Evaluate → Deploy → Iterate │ │ └─── The lifecycle is iterative — teams cycle back to earlier phases as feedback arrives, not just run it once │ ├─── A2. Iteration as a Core Property │ │ ├─── Gen AI Lifecycle is iterative — continuously refined, never a single pass │ │ └─── Iteration is triggered by evaluation results, user feedback, and changing requirements │ ├─── A3. Capabilities of Generative AI │ │ ├─── Adaptability — Works across multiple domains and task types without redesign │ │ ├─── Responsiveness — Generates outputs in real time, supporting interactive use cases │ │ ├─── Simplicity — Automates workflows that previously required multiple manual steps │ │ ├─── Creativity — Produces novel ideas, phrasing, or content rather than retrieving fixed answers │ │ ├─── Data Efficiency — Learns generalizable patterns from large-scale training data │ │ ├─── Personalization — Tailors outputs to an individual user's context or preferences │ │ └─── Scalability — Generates content at a volume no human team could match │ └─── A4. Challenges of Generative AI │ ├─── Regulatory Violations — Misuse of sensitive, restricted, or regulated data in training or inference │ ├─── Social Risks — Outputs that are harmful, misleading, or reinforce bias at scale │ ├─── Privacy Concerns — Exposure of personal or confidential information through generated output │ ├─── Toxicity — Offensive, unsafe, or inappropriate generated content │ ├─── Hallucinations — Plausible-sounding but factually incorrect output presented with confidence │ ├─── Interpretability — Difficulty explaining why a model produced a specific output │ └─── Nondeterminism — Inconsistent output for the same input across separate runs ├─── B. Defining a Use Case │ ├─── B1. Why Use Case Definition Comes First │ │ ├─── A bad use case definition produces a failed AI project, regardless of model quality │ │ └─── Use case definition also determines the success metrics used later in Evaluation │ ├─── B2. Business Use Case Structure │ │ ├─── Use Case Name — A short, descriptive identifier for the use case │ │ ├─── Brief Description — A one-paragraph summary of the objective │ │ ├─── Actors — The users or systems that interact with the solution │ │ ├─── Preconditions — Conditions that must be true before the flow can start │ │ ├─── Basic Flow — The main, step-by-step success path │ │ ├─── Alternative Flows — Error handling and edge-case paths │ │ ├─── Postconditions — The system's final state after successful execution │ │ ├─── Business Rules — Policies and constraints the solution must respect │ │ ├─── Nonfunctional Requirements — Performance, security, and usability requirements │ │ ├─── Assumptions — Conditions expected to hold true in the environment │ │ └─── Notes — Additional clarifications that don't fit elsewhere │ ├─── B3. Key Metrics for Generative AI Use Cases │ │ ├─── Cost Savings — Reduction in operational expense versus the current process │ │ ├─── Time Savings — Faster completion of the target task │ │ ├─── Quality Improvement — Measurably better outputs than the current baseline │ │ ├─── Customer Satisfaction — Improved experience, typically tracked via survey or NPS │ │ └─── Productivity Gains — Increased throughput or efficiency per team member │ └─── B4. Common Approaches │ ├─── Process Automation — Automating repetitive, rules-based tasks │ ├─── Decision Support — Assisting — not replacing — human decision-making │ ├─── Personalization — Tailoring experiences to individual users │ ├─── Content Generation — Creating text, images, or other media │ └─── Innovation Exploration — Generating new ideas or design directions ├─── C. Selecting a Foundation Model │ ├─── C1. What a Foundation Model Is │ │ ├─── An FM is the pre-trained base every Gen AI application builds on │ │ └─── No single FM is universally best — selection is a trade-off exercise, not a leaderboard lookup │ ├─── C2. Key Selection Criteria │ │ ├─── Cost — Compute expense, licensing fees, and customization cost │ │ ├─── Modality — Text, image, audio, or multimodal input/output support │ │ ├─── Latency — How fast the model must respond for the use case │ │ ├─── Multilingual Support — Which languages the model handles well │ │ ├─── Model Size — Trade-off between performance and resource consumption │ │ ├─── Model Complexity — Capability versus difficulty of deployment and operation │ │ ├─── Customization Ability — Whether and how well the model supports fine-tuning │ │ ├─── Input/Output Length — Context window and generation length limitations │ │ ├─── Responsibility — Bias, safety, and ethical behavior of the model │ │ └─── Deployment Compatibility — Ease of integration into the target production environment │ └─── C3. Example Models on Amazon Bedrock │ ├─── AI21 Jurassic-2 — Optimized for text generation and summarization tasks where fluent, on-topic prose matters most. │ ├─── Amazon Titan — AWS's own model family, offering embeddings alongside text and image generation — useful when the pipeline also needs vector search. │ ├─── Anthropic Claude — Strong at reasoning, multi-turn chat, and coding tasks, making it a common choice for assistant-style and developer-facing applications. │ ├─── Cohere Command — Tuned for a broad range of natural language processing (NLP) tasks such as classification and extraction. │ ├─── Meta Llama — A general-purpose large language model (LLM) family suited to a wide range of tasks without heavy specialization. │ └─── Mistral Large — Positioned for high reasoning capability, competing with larger models at a smaller footprint. ├─── D. Improving Foundation Model Performance │ ├─── D1. Prompt Engineering: Core Techniques │ │ ├─── Design — Writing clear, structured prompts with explicit instructions │ │ ├─── Augmentation — Adding examples or constraints to narrow the model's behavior │ │ ├─── Tuning — Iteratively refining a prompt based on observed output quality │ │ ├─── Ensembling — Combining multiple prompts or prompt variants for a stronger result │ │ └─── Mining — Systematically discovering prompt phrasings that perform well │ ├─── D2. Common Prompting Methods │ │ ├─── Zero-shot — Asking the model to perform a task with no example provided │ │ ├─── Few-shot — Providing a small number of examples inside the prompt to guide format and style │ │ ├─── Chain-of-thought (CoT) — Prompting the model to reason step-by-step before answering │ │ ├─── Tree-of-thought (ToT) — Exploring multiple reasoning branches before selecting the best path │ │ ├─── Self-consistency — Sampling multiple reasoning paths and taking the majority answer │ │ └─── ReAct — Interleaving reasoning steps with tool-use actions │ ├─── D3. Retrieval Augmented Generation (RAG) │ │ ├─── RAG = Retrieval System + Generative Model │ │ └─── RAG directly targets the hallucination challenge from Section A by grounding answers in retrieved facts │ ├─── D4. RAG Business Applications │ │ ├─── Question-Answering Systems — Answering user questions using an organization's own documents │ │ ├─── Knowledge Base Enhancement — Surfacing accurate information from large internal knowledge bases │ │ └─── Content Generation — Producing content that must stay factually grounded in source material │ ├─── D5. Fine-Tuning │ │ ├─── Fine-tuning methods: Instruction Tuning, RLHF │ │ └─── Process: pre-trained model → prepare dataset → add task-specific layers → train → evaluate & iterate │ ├─── D6. Model from Scratch & the Trade-Off Summary │ │ ├─── Pre-trained — Lowest cost │ │ ├─── Pre-trained — Least control over model behavior │ │ ├─── Fine-tuned — Balanced cost and control │ │ ├─── Fine-tuned — Requires a labeled domain dataset │ │ ├─── From Scratch — Highest cost │ │ └─── From Scratch — Highest customization and accuracy ceiling │ └─── D7. Agents for Automation │ ├─── Task Coordination — Sequencing and managing multi-step workflows │ ├─── Logging and Reporting — Recording actions taken and outcomes produced │ ├─── Scalability — Handling many concurrent workflow instances │ └─── System Integration — Connecting to external tools, APIs, and data sources ├─── E. Evaluating Foundation Models │ ├─── E1. Evaluation Methods │ │ ├─── Human Evaluation — Qualitative judgment from real reviewers — the highest-quality signal, but also the most costly and slowest to run at scale. │ │ ├─── Benchmark Datasets — Standardized datasets that allow apples-to-apples comparison between different models or configurations. │ │ └─── Automated Metrics — Fast, scalable scoring that can run continuously, but is not a substitute for human judgment on nuanced quality. │ ├─── E2. Benchmark Examples │ │ ├─── GLUE — General Language Understanding Evaluation — a suite of core NLP understanding tasks │ │ ├─── SuperGLUE — A harder successor to GLUE, designed as earlier models saturated it │ │ ├─── SQuAD — Stanford Question Answering Dataset — benchmarks reading-comprehension QA │ │ └─── WMT — Workshop on Machine Translation — benchmarks translation quality │ ├─── E3. Key Metrics │ │ ├─── ROUGE — Measures overlap between generated and reference text — used for summarization quality │ │ ├─── BLEU — Measures translation quality by comparing generated output to reference translations │ │ └─── BERTScore — Measures semantic similarity using contextual embeddings, catching paraphrases that ROUGE/BLEU miss │ └─── E4. Additional Metrics │ ├─── Perplexity: prediction confidence (lower = better) │ └─── F1 Score: balances precision and recall for classification-style outputs └─── F. Deployment of Generative AI Applications ├─── F1. Key Deployment Considerations │ ├─── Cost — Pay-per-use resource pricing that scales with actual traffic │ ├─── Regions — Service availability differs by AWS region — latency and compliance both depend on this │ ├─── Quotas — Built-in resource limits that must be checked against expected load │ └─── Security — Shared responsibility between AWS and the user, plus risks specific to external/third-party models └─── F2. Deployment Goals ├─── Scalability — Handling growth in traffic and usage without redesign ├─── Reliability — Consistent availability and correct behavior over time ├─── Security — Protecting data, access, and model behavior from misuse └─── User Experience — Making the system fast, clear, and usable for its actual audience

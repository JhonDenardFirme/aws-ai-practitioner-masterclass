Module 6. Optimizing Foundation Models ├─── A. Foundations of FM Optimization │ ├─── A1. What Optimization Means │ │ ├─── FM Optimization = improving output quality and relevance post-selection │ │ └─── The target is business alignment, not abstract benchmark performance │ └─── A2. Core Objectives │ ├─── Embedding Storage — Store vector representations of data in a database for fast retrieval │ ├─── Agents — Automate multi-step workflows that connect the model to backend systems │ ├─── Evaluation — Measure how effectively the FM performs against business goals │ └─── Fine-Tuning — Adapt the model's weights to a specific domain or task ├─── B. Business Case: AnyCompany Telecom Chatbot │ ├─── B1. Problem & Goals │ │ └─── Goal: reduce tickets by 70%, raise satisfaction to ≥ 4/5 │ ├─── B2. ML Challenges │ │ ├─── Generic models lack domain knowledge by default │ │ └─── Two needs: enterprise data integration + task automation capability │ └─── B3. Solution Components │ ├─── Knowledge Base — Built from chat logs, support tickets, and call recordings — the raw material a RAG system retrieves from. │ ├─── Data Preparation — Collection, cleaning, and anonymization of that raw material before it's usable — anonymization matters here since the source data includes real customer conversations. │ └─── Automation — Agents that can modify user plans and execute backend actions, turning a conversation into a completed task. ├─── C. Retrieval-Augmented Generation (RAG) │ ├─── C1. RAG and Enterprise Data Integration │ │ ├─── RAG's enterprise data sources: documents, reports, manuals │ │ └─── RAG = context injection without retraining — the module's core definition │ ├─── C2. Vector Embeddings │ │ ├─── Embedding = numerical vector representation capturing semantic meaning │ │ └─── Similar meaning → similar vectors (Sea ≈ Ocean; Stapler ≠ Ocean) │ ├─── C3. Vector Databases & Similarity Search │ │ ├─── Vector DB purpose: store embeddings, enable fast similarity search, retrieve relevant context │ │ └─── Two search techniques: k-NN (nearest vector search), Cosine Similarity (angle-based) │ └─── C4. AWS Vector Database Options │ ├─── Amazon OpenSearch — Available in Provisioned and Serverless modes for vector search │ ├─── pgvector — A PostgreSQL extension available on Amazon RDS and Aurora │ └─── Amazon Kendra — An enterprise search service usable as a managed retrieval layer ├─── D. Agents in Generative AI Systems │ ├─── D1. Key Functions of Agents │ │ ├─── Intermediary Operations — Connects the model to backend systems, translating a model's decision into a system-level call. │ │ ├─── Action Execution — Actually performs actions — modifying accounts, triggering workflows — rather than just describing what should happen. │ │ └─── Feedback Integration — Feeds outcomes of those actions back into the system, improving the model's behavior over time. │ └─── D2. AnyCompany Agent Example │ ├─── Agent 1 — Manages customer plans — upgrades, downgrades, modifications │ ├─── Agent 2 — Updates the knowledge base as new information becomes available │ └─── Agent 3 — Collects feedback from completed interactions ├─── E. Evaluating Generative AI Systems │ ├─── E1. Evaluation Methods │ │ ├─── Human Evaluation — Focuses on user experience, relevance, and creativity — its strength is capturing nuance no automated metric detects, though it's slow and expensive to scale. │ │ ├─── Benchmark Datasets — Provide quantitative evaluation across accuracy, speed, and scalability — built from SME-written questions, identified context, and defined ground-truth answers. │ │ └─── LLM-as-Judge — Uses another model to compare a generated answer against a reference answer, scoring accuracy, relevance, and completeness automatically. │ └─── E2. Benchmark Creation Steps │ └─── Benchmark creation order: Questions (by SMEs) → Context → Answers (ground truth) ├─── F. Business Case: AnyCompany Retail Personalization │ ├─── F1. Problem & Business Metrics │ │ ├─── Conversion Rate — Share of visits that result in a completed purchase │ │ ├─── Average Order Value — Average revenue per completed order │ │ └─── Customer Retention — Rate at which customers return for repeat purchases │ └─── F2. AI Solution Components │ ├─── Fine-Tuned Model — Trained on the company's own transaction data and user behavior, rather than generic retail knowledge. │ ├─── Recommendation Engine — Uses the fine-tuned model's understanding of behavior to generate personalized product suggestions. │ └─── Feedback Loop — Feeds purchase outcomes back into the system, enabling continuous improvement of recommendations. ├─── G. Fine-Tuning Foundation Models │ ├─── G1. Why Fine-Tune: Benefits │ │ ├─── Increased Specificity — Outputs speak the domain's actual vocabulary and conventions │ │ ├─── Improved Accuracy — Fewer errors on domain-specific tasks versus a generic model │ │ ├─── Reduced Bias — Domain-curated data can correct skew present in general training data │ │ └─── Enhanced Efficiency — Shorter prompts needed since domain knowledge is baked into the weights │ └─── G2. Fine-Tuning Approaches │ ├─── Instruction Tuning — Trains the model to follow structured input-output examples │ ├─── RLHF — Optimizes model behavior using human feedback on output quality │ ├─── Domain Adaptation — Uses specialized, domain-specific datasets for retraining │ ├─── Transfer Learning — Reuses knowledge already captured in a pretrained model as a starting point │ └─── Continuous Pretraining — Updates a model's knowledge with new data over time, without a full retrain ├─── H. Data Preparation for Fine-Tuning │ ├─── H1. Initial Training Data vs. Fine-Tuning Data │ │ ├─── Initial training data: large-scale, diverse, general-purpose │ │ └─── Fine-tuning data: specific, relevant, high-quality — small and curated, not broad │ └─── H2. Key Data Preparation Steps │ ├─── Data Curation — Selecting and assembling the dataset from raw sources │ ├─── Labeling — Attaching correct target outputs or annotations to each example │ ├─── Governance & Compliance — Ensuring data use meets legal and regulatory requirements │ ├─── Bias Checking — Identifying and correcting skew that would propagate into the fine-tuned model │ └─── Feedback Integration — Incorporating real usage feedback back into future data preparation cycles └─── I. Model Evaluation Metrics └─── I1. ROUGE, BLEU, and BERTScore ├─── ROUGE — Measures overlap with a reference text ├─── ROUGE — Focuses on recall — completeness of coverage ├─── BLEU — Measures precision of N-grams against a reference ├─── BLEU — Includes a brevity penalty to discourage overly short output ├─── BERTScore — Measures semantic similarity using contextual embeddings └─── BERTScore — Captures meaning even when exact wording differs

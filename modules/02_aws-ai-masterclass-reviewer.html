Module Introduction

Module 1 established what AI, ML, DL, and Generative AI mean and how the underlying model architectures work. This module answers a different question: where does all of that actually get used? It surveys real AI use cases across six industries, breaks down the four core application domains that implement AI capability into working systems, and gives a practical framework for deciding when ML is the right tool at all.

The second half returns to generative AI specifically — its capabilities, its risks, how to select a Foundation Model for a given task, and how to measure whether a deployed system is actually delivering business value. Together, this module is the bridge between Module 1's conceptual foundation and the hands-on technique modules that follow.

A AI Overview and Definitions A
A1AI, ML, DL, and Generative AI Recap

As covered in Module 1, four nested fields describe increasingly specific AI capability: Artificial Intelligence (systems mimicking human intelligence), Machine Learning (learning patterns from data), Deep Learning (neural network-based learning), and Generative AI (creating new data or content).

Generative AI creates — ML predicts and analyzes. This module builds on that distinction by walking through concrete, real-world use cases across industries.

Key Points
  • Generative AI creates; ML predicts/analyzes — the same core distinction from Module 1, applied here to real use cases

Audio

Code
  • Programming outputs
Key Points
  • Four content types: Text, Visual, Audio, Code

D Machine Learning Fundamentals in Practice D
D1When ML Is Appropriate

ML earns its complexity cost under three conditions — spam filtering is the module's example of a problem that satisfies all three at once.

Complex Rules
  • The correct logic is too difficult to define manually
High Variability
  • Many influencing factors make a fixed rule set brittle
Large Scale
  • The problem involves massive datasets no manual process could handle
Key Points
  • Use ML when rules are too complex or scale is too high — the module's core decision rule
  • Spam filtering satisfies all three conditions: complex, high-variability, large-scale
D2When ML Is NOT Appropriate

ML is the wrong tool when Simple Logic already provides clear, deterministic rules, or when a Low-Complexity Task is already solvable with predefined steps — basic calculations being the module's example. Do not use ML if a rule-based solution already works: ML adds cost, unpredictability, and maintenance burden that a working deterministic solution doesn't need.

Key Points
  • Do not use ML if a rule-based solution already works — the module's core caution
E Machine Learning Techniques and Use Cases E
E1Supervised Learning: Classification vs. Regression

Supervised Learning learns from labeled data to predict outputs, and splits into two types. Classification predicts a categorical outcome — used for fraud detection, image classification, customer retention, and diagnostics. Regression predicts a continuous value — used for weather forecasting, market trends, life expectancy, and population growth.

Key Points
  • Classification = categorical prediction (fraud detection, image classification, customer retention, diagnostics)
  • Regression = continuous prediction (weather forecasting, market trends, life expectancy, population growth)
E2Unsupervised Learning: Clustering vs. Dimensionality Reduction

Unsupervised Learning learns from unlabeled data to discover hidden patterns. Clustering groups similar data points — used for customer segmentation, targeted marketing, and recommendation systems. Dimensionality Reduction simplifies data by reducing the number of features — used for data visualization, compression, and feature extraction.

Unsupervised learning has no labels — the model creates its own structure from the data, rather than being told what the correct groupings or reduced representations should be.

Key Points
  • Clustering: customer segmentation, targeted marketing, recommendation systems
  • Dimensionality Reduction: data visualization, compression, feature extraction
E3Reinforcement Learning

Reinforcement Learning learns optimal actions through trial-and-error, using three core components. AWS DeepRacer — autonomous driving — is the module's concrete example.

Agent
  • The decision-maker taking actions
Environment
  • The space the agent interacts with
Feedback
  • Rewards and penalties that shape future decisions
Key Points
  • RL learns optimal actions through trial-and-error — the module's key insight
  • RL components: Agent, Environment, Feedback (rewards/penalties)

H Challenges of Generative AI H
H1Seven Risks

The same generative capability that creates business value also introduces seven distinct risk categories that must be managed before deployment.

Regulatory Violations
  • Exposure of sensitive or restricted data
Social Risks
  • Harmful or reputationally damaging content
Privacy Concerns
  • Misuse of personal data
Toxicity
  • Offensive or inappropriate outputs
Hallucinations
  • Plausible but false generated information
Interpretability
  • Difficulty understanding why an output was produced
Nondeterminism
  • Inconsistent outputs across runs
Key Points
  • Hallucinations = plausible but false outputs — the module's key definition
  • AI outputs must always be verified — the module's core caution
H2Mitigation Strategies

Seven mitigation strategies map roughly one-to-one against the seven risks above, though several address multiple risks at once.

Data Anonymization
  • Protects sensitive data from exposure
Model Testing
  • Evaluates behavior before deployment
Security Measures
  • Encryption and firewalls protecting the system
Data Curation
  • Removes harmful content from training data
Guardrails
  • Filters outputs before they reach the user
Verification
  • Cross-checks outputs against trusted sources
Monitoring
  • Detects inconsistencies in ongoing production use
Key Points
  • Seven mitigations: Data Anonymization, Model Testing, Security Measures, Data Curation, Guardrails, Verification, Monitoring

J Business Metrics for Generative AI J
J1Five Business Metrics

These five metrics are how a business actually knows whether a deployed generative AI system is working, beyond raw model accuracy.

User Satisfaction
  • Direct feedback on the experience
Average Revenue Per User (ARPU)
  • Revenue generated per user
Cross-Domain Performance
  • Effectiveness across multiple use cases at once
Conversion Rate
  • Rate of desired action completion
Efficiency
  • Resource and time optimization achieved
Key Points
  • Five business metrics: User Satisfaction, ARPU, Cross-Domain Performance, Conversion Rate, Efficiency
J2Key Insights on Metrics

Metrics depend on the use case — there is no universal metric set that applies to every deployment. The purpose of measurement is always to measure ROI and guide optimization decisions going forward.

Success = technical performance + business value — a model can be technically excellent and still fail the organization if its outputs don't move a metric the business actually cares about.

Key Points
  • Metrics are use-case-dependent, not universal
  • Success = technical performance + business value — the module's closing insight
Module Summary
  • AI use cases span at least six industries — Media, Retail, Healthcare, Life Sciences, Financial Services, Manufacturing — and AI creates business value specifically through automation and optimization.
  • Four core application domains implement AI capability into working systems: Computer Vision, NLP, Intelligent Document Processing, and Fraud Detection.
  • Use ML when rules are too complex or scale is too high; don't use ML when a simple, deterministic, rule-based solution already works.
  • Supervised Learning (Classification, Regression), Unsupervised Learning (Clustering, Dimensionality Reduction), and Reinforcement Learning (Agent, Environment, Feedback) each solve a distinct category of problem.
  • Generative AI's seven capabilities create value, but its seven risks (bias, hallucination, privacy, toxicity, and more) require matched mitigation strategies; model selection balances task fit, cost, and performance, and ultimate success depends on both technical performance and business value.

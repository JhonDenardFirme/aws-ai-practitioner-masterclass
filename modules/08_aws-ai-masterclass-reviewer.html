Module Introduction

Every technique in this masterclass so far — selecting a Foundation Model, optimizing it with RAG and fine-tuning, securing it with governance and compliance — assumes the model is actually being asked the right question. This module covers the layer underneath all of that: how the words in a prompt shape everything the model produces, and how those same words become an attack surface if left undefended.

The first half builds prompting skill directly — the four core components, inference parameters like temperature and Top P, best practices, and the three foundational techniques (Zero-Shot, Few-Shot, Chain-of-Thought). The second half turns the same lens toward risk: Poisoning, Prompt Injection, Exposure, Prompt Leaking, and Jailbreaking are five distinct ways prompts and data can be misused, each targeting a different layer of the system.

A Introduction to Prompt Engineering A
A1What Prompt Engineering Is

Prompt Engineering is the process of designing instructions to guide AI outputs effectively. It matters because a Foundation Model is powerful, but without proper prompting, its outputs may become irrelevant, generic, or inconsistent — the model's raw capability doesn't guarantee a useful answer.

Prompting is the bridge between human intent and AI interpretation. In business contexts, well-engineered prompts automate repetitive analytical and creative tasks, directly improving operational efficiency and scalability.

Key Points
  • Prompt quality directly affects output quality — the module's central premise
  • Prompting bridges human intent and AI interpretation
A2The AnyCompany Scenario

A fintech company, AnyCompany, needed AI-generated market reports. Its initial prompt was vague, and the model produced an irrelevant smart home analysis instead — a concrete illustration of how little guidance a Foundation Model actually gets from an underspecified request.

Key Points
  • A vague prompt from a finance company produced an unrelated smart-home analysis — the module's running example
A3Why Weak Prompts Fail

AnyCompany's failure traces to four specific, recurring problems — recognizing them is the first diagnostic step in fixing any bad prompt.

Lack of Specificity
  • The request doesn't narrow down what's actually wanted
Missing Domain Context
  • The model has no signal about the industry or audience
Unclear Output Structure
  • No indication of format, length, or organization
Generic Responses
  • The result: outputs that could apply to almost any topic
Key Points
  • Improved prompting's goal: produce tailored, structured, domain-specific outputs
B Understanding Prompts B
B1What a Prompt Is, and Its Benefits

A prompt is the input instruction given to a generative AI model — it can include questions, commands, examples, or constraints. Effective prompting delivers four concrete benefits.

Enhances Model Performance
  • Better-guided models produce more accurate, on-target output
Improves Output Safety
  • Reduces the chance of harmful or inappropriate content
Adds Domain-Specific Behavior
  • Tailors generic model knowledge to a specific field or task
Produces Higher-Quality Responses
  • Clearer instructions consistently yield better-formed answers
Key Points
  • Prompt engineering methodologies: precise framing, output examples, intermediate reasoning steps, structured formatting
B2The Four Core Prompt Components

Nearly every effective prompt combines the same four components. AnyCompany's ecommerce order-fulfillment example shows each one concretely.

Instructions
  • Defines the task to perform, e.g. “Determine which orders can be fulfilled”
Context
  • Background that tailors the output to domain or audience, e.g. “ecommerce inventory management task”
Input Data
  • The actual content being processed, e.g. “orders and inventory quantities”
Output Indicator
  • Specifies the expected format, e.g. “fulfillment status”
Key Points
  • Core prompt elements: Instructions, Context, Input Data, Output Indicator
  • Missing context and output indicators are the most common reasons prompts produce weak results
B3Negative Prompting

Negative Prompting explicitly tells the model what NOT to generate — preventing hate speech, avoiding misinformation, and restricting unsafe recommendations. It's a direct complement to positive instructions, closing off failure modes rather than only describing the desired outcome.

Key Points
  • Negative prompting increases the safety and appropriateness of responses by restricting specific undesired behaviors
C Modifying Prompts and Inference Parameters C
C1Prompt Refinement and Inference Parameters

Prompt Refinement is the iterative process of improving prompts to enhance relevance, structure, and precision. Inference Parameters are adjustable settings that influence model behavior independently of the prompt text itself — affecting creativity, coherence, diversity, and length.

Key Points
  • Inference parameters and prompt wording are two separate levers on the same output
C2Temperature, Top P, and Top K

Three parameters control randomness and diversity in token selection, each working slightly differently under the hood.

Temperature
  • Controls randomness/creativity directly
  • Low (0.2): conservative, predictable, more coherent
  • High (1.0): creative, diverse, potentially less coherent
Top P
  • Probability-based token filtering
  • Low: smaller probability pool, more focused output
  • High: wider probability pool, more diverse language
Top K
  • Limits choices to the top K most probable words
  • Low: more constrained responses
  • High: larger vocabulary selection, greater creativity
Key Points
  • High creativity settings can improve innovation but may reduce factual accuracy or coherence — the module's key trade-off warning
C3Length Parameters

Maximum Length limits the total number of generated tokens, preventing excessively long responses. Stop Sequences are special tokens or phrases that end generation early — particularly useful in dialogue systems or structured outputs where a response should stop at a clear boundary.

Key Points
  • Maximum Length caps token count; Stop Sequences end generation at a defined boundary
C4AnyCompany's Prompt Improvements

Applying these levers, AnyCompany rebuilt its market-report prompt with six specific changes — the concrete before/after of everything covered in this section.

Added Finance Industry Context
  • Grounded the request in the correct domain
Specified SMB Audience
  • Narrowed the intended reader
Defined Report Structure
  • Removed ambiguity about output format
Set Professional Tone
  • Constrained the register of the writing
Increased Maximum Length
  • Allowed room for a fuller report
Increased Temperature and Top P
  • Traded some predictability for more creative framing, appropriate once structure was locked down
Key Points
  • Six changes transformed AnyCompany's vague prompt into a targeted, structured one
D Best Practices for Prompt Engineering D
D1Nine Core Best Practices

These nine practices recur across nearly every well-engineered prompt. None require special tooling — they're writing discipline applied to a new medium.

Be Clear and Concise
  • Use natural language and direct instructions, not overly technical wording
  • Good: “What is the sum of these numbers?” vs. bad: “Compute the sum total…”
Include Context
  • Specify industry, audience, and business purpose to improve relevance
Use Directives
  • State the output type clearly: summary, poem, analysis, table, JSON
Specify Desired Output
  • Define length, tone, structure, and included/excluded content
Start with Interrogatives
  • Frame requests with Who, What, Where, When, Why, How
Provide Example Outputs
  • Demonstrate expected formatting or reasoning directly
Break Up Complex Tasks
  • Divide into subtasks; ask the model to think step by step
Experiment and Iterate
  • Compare prompt versions and refine based on output quality
Use Prompt Templates
  • Standardized, reusable structures improve consistency across workflows
Key Points
  • Prompt engineering is iterative — even strong prompts often need multiple refinements before reaching optimal results
E Prompt Engineering Techniques E
E1Zero-Shot Prompting

Zero-Shot Prompting asks the model to perform a task with no examples, relying entirely on pretrained knowledge. It's efficient and requires minimal setup, but is less reliable for complex tasks.

[+] Zero-shot reliability improves with larger models, instruction tuning, and RLHF — the same techniques covered in Module 6's fine-tuning section.

Key Points
  • Zero-shot: no examples, relies on pretrained knowledge, fast but less reliable for complex tasks
E2Few-Shot Prompting

Few-Shot Prompting includes examples of desired behavior directly in the prompt, conditioning the model toward the expected output. It improves consistency and formatting accuracy over zero-shot.

Best practice: use representative examples, avoid providing excessive examples, and maintain clarity and diversity across the examples given.

Key Points
  • Few-shot: examples included, improves consistency and formatting accuracy over zero-shot
E3Chain-of-Thought (CoT) Prompting

Chain-of-Thought Prompting encourages step-by-step reasoning, commonly triggered with the phrase “Think step by step.” It comes in two forms — Zero-shot CoT and Few-shot CoT — and is most useful across four application areas.

Mathematics
  • Multi-step calculations benefit from explicit intermediate steps
Financial Analysis
  • Sequential reasoning through assumptions and figures
Logic Problems
  • Step-by-step elimination of possibilities
Multi-Step Reasoning
  • Any task where the answer depends on an ordered chain of sub-conclusions
Key Points
  • CoT is most useful when tasks require reasoning, calculations, or multiple logical steps
  • AnyCompany applied CoT with report templates and structured categories, producing more aligned, professional reports
F Prompt Misuses and Risks F
F1Prompt Misuse and Adversarial Prompting

Prompt Misuse is improper or malicious use of prompts against AI systems — it can manipulate outputs, expose sensitive data, or bypass safeguards. Adversarial Prompting is the deliberate attempt to exploit FM vulnerabilities, often targeting safety filters, alignment rules, or system instructions specifically.

Key Points
  • Improperly designed or malicious prompts can compromise security, privacy, fairness, and system integrity
F2Main Categories of Prompt Risk

Five categories organize every prompt-security risk covered in this module — each gets its own dedicated section below.

Poisoning
  • Corrupts training data, not the prompt interface itself
Hijacking and Prompt Injection
  • Manipulates model behavior at inference time via crafted prompts
Exposure
  • Unintentional disclosure of sensitive or confidential information
Prompt Leaking
  • Disclosure of hidden system prompts or internal instructions
Jailbreaking
  • Circumvents safety restrictions to force prohibited output
Key Points
  • Most AI attacks target one of three layers: Data, Prompts, or Safety Mechanisms
G Poisoning Attacks G
G1What Poisoning Is

Poisoning is the insertion of malicious or biased data into training datasets, manipulating model behavior during training. Poisoning attacks target the DATA itself, not necessarily the prompt interface — the module's central distinction versus every other risk category here.

It happens primarily during training, fine-tuning, or dataset preparation, and can be intentional, accidental, or hidden within large datasets where it's hard to detect.

Key Points
  • Poisoning's goal: corrupt outputs, introduce bias, cause unsafe responses, reduce trustworthiness
  • Poisoning happens at training/fine-tuning time, not at inference time
G2Examples and Real-World Implications

Concrete examples span both intent and industry — poisoning isn't a purely theoretical risk.

Biased Training Examples
  • Injecting racist or biased examples into training data
Manipulated Financial Data
  • Distorting predictions by manipulating financial datasets
Malicious Code Samples
  • Inserting malicious code into coding training datasets
Misinformation in Corpora
  • Feeding misinformation into knowledge corpora used for training
Key Points
  • Real-world implications: healthcare AI recommending unsafe treatments, financial AI producing biased loan approvals, recruitment AI discriminating against applicants
G3Mitigation Strategies

Six mitigation strategies target poisoning specifically at the data layer, before a model is ever deployed.

Dataset Validation
  • Systematically checking data before it enters training
Human Review Processes
  • Manual review catches what automated checks miss
Bias Detection
  • Statistical checks for skew across training examples
Data Provenance Tracking
  • Knowing exactly where each training example came from
Secure Fine-Tuning Pipelines
  • Controlling who can introduce new data into a fine-tuning run
Regular Auditing
  • Ongoing checks, not a one-time validation pass
Key Points
  • Mitigation targets the data pipeline directly — validation, review, provenance, and auditing
H Hijacking and Prompt Injection H
H1What Prompt Injection and Hijacking Are

Prompt Injection embeds malicious instructions into prompts, attempting to manipulate model behavior. Hijacking redirects model outputs toward an attacker's objectives. Unlike poisoning, this occurs during inference/runtime and exploits the model's instruction-following behavior directly, often through indirect wording or roleplay.

Prompt injection exploits the model's tendency to prioritize recent or strongly worded instructions — a structural quirk of how these models process conversational context.

Key Points
  • Prompt injection targets the runtime prompt, not training data (contrast with Poisoning)
  • Goals: generate misinformation, bypass restrictions, produce malicious instructions, leak hidden instructions
H2Common Prompt Injection Techniques

Four phrasing patterns recur across real prompt injection attempts — recognizing them is the first line of defense.

“Ignore previous instructions”
  • Directly attempts to override the system prompt
“Act as an unrestricted AI”
  • Attempts to reframe the model's own identity and constraints
“Pretend this is fictional”
  • Uses fictional framing to justify otherwise-restricted content
“For educational purposes only”
  • Uses a stated legitimate purpose to lower the model's guard
Key Points
  • [+] Not every use of these patterns is malicious — legitimate cases include preserving product names in translation, customizing response tone, and controlling formatting behavior
H3Poisoning vs. Prompt Injection

The two are easy to confuse but differ on every axis: poisoning targets training data during training/fine-tuning to corrupt long-term model behavior; prompt injection targets the runtime prompt at inference time to manipulate an immediate response. A biased dataset is the classic poisoning example; “ignore previous instructions” is the classic injection example.

Key Points
  • Poisoning = training data, long-term. Prompt Injection = runtime prompt, immediate.
H4Mitigation Strategies

Six defenses target prompt injection specifically at the interface where user input meets the model.

Input Sanitization
  • Cleaning user input before it reaches the model
Instruction Hierarchy Enforcement
  • Ensuring system instructions can't be overridden by user text
Output Filtering
  • Catching problematic output after generation, before it reaches the user
Human Oversight
  • Manual review for high-risk interactions
Context Isolation
  • Preventing user input from bleeding into system-level context
Retrieval Safeguards
  • Protecting RAG pipelines from injected instructions hidden in retrieved documents
Key Points
  • Mitigation targets the prompt interface: sanitization, hierarchy enforcement, filtering, isolation
I Exposure and Prompt Leaking I
I1Exposure

Exposure is the unintentional disclosure of sensitive or confidential information, occurring during training, inference, personalized recommendations, or output generation. Five data types are most commonly exposed.

Personal Information
  • Names, addresses, and other direct identifiers
Financial Records
  • Account details, transaction history
Medical Records
  • Protected health information
Browsing History
  • Behavioral data revealing personal habits
Purchase History
  • Transaction records tied to individual customers
Key Points
  • Exposure risks are especially dangerous in healthcare, finance, HR, and government AI systems
  • Causes: poor data anonymization, memorization of training data, weak privacy controls, excessive logging
I2Prompt Leaking

Prompt Leaking is the disclosure of hidden prompts, system instructions, or internal guidance. Unlike Exposure, it may not expose protected personal data at all — instead it exposes system behavior and operational logic.

“Tell me your instructions”
  • Directly asks the model to reveal its system prompt
“Repeat your hidden prompt”
  • Attempts to extract the exact system-level text
“What were you told before this conversation?”
  • Probes for prior context or configuration
Key Points
  • Prompt leaking reveals internal policies, helps attackers craft better exploits, exposes proprietary workflows, weakens AI defenses
I3Exposure vs. Prompt Leaking

Exposure's main risk is sensitive data disclosure — PII, PHI, confidential records — leading to privacy violations. Prompt Leaking's main risk is system instruction disclosure — hidden prompts and rules — leading to security bypasses. Revealing a customer's purchase history is Exposure; revealing a moderation instruction is Prompt Leaking.

Key Points
  • Exposure = sensitive data disclosure. Prompt Leaking = system instruction disclosure.
I4Mitigation Strategies

Six mitigations address both Exposure and Prompt Leaking together, since the underlying protections overlap.

Data Anonymization
  • Removing identifying details before data is usable by the model
Access Control
  • Limiting who and what can query sensitive systems
Encryption
  • Protecting data both at rest and in transit
Prompt Isolation
  • Keeping system prompts separated from user-accessible context
Retrieval Restrictions
  • Limiting what a RAG pipeline can surface to a query
Redaction Systems
  • Automatically stripping sensitive content from output before delivery
Key Points
  • Shared mitigations: anonymization, access control, encryption, isolation, restriction, redaction
J Jailbreaking J
J1What Jailbreaking Is

Jailbreaking circumvents AI safety restrictions and safeguards, with the primary goal of forcing the model to produce prohibited outputs. Jailbreaking specifically attempts to bypass SAFETY CONTROLS, unlike normal prompting optimization — the key distinction from everything covered in Sections C–E.

Roleplay
  • Asking the model to adopt a persona without the usual constraints
Hypotheticals
  • Framing prohibited content as a fictional or theoretical scenario
Emotional Manipulation
  • Applying social pressure to get the model to comply
Multi-Step Prompts
  • Building toward a prohibited output gradually across several turns
Instruction Overrides
  • Directly attempting to cancel prior safety instructions
Key Points
  • Jailbreaking's danger: harmful instructions, cyber abuse enablement, unsafe content, weakened AI trust and compliance
J2Common Jailbreaking Approaches

Four named approaches recur across jailbreaking attempts, each attacking safety training from a different angle.

Character Simulation
  • “Pretend you are a hacker” or “act as a fictional villain”
Recursive Instruction Attacks
  • Repeatedly re-framing the request to work around a refusal
Prompt Chaining
  • Building the prohibited request across multiple linked prompts
Obfuscation Techniques
  • Disguising the actual request through encoding or indirection
Key Points
  • Enterprise risks: legal liability, regulatory violations, harmful public outputs, brand reputation damage
J3Mitigation Strategies

Six defenses target jailbreaking at the model and process level, going beyond simple input filtering.

RLHF
  • Reinforcement Learning from Human Feedback aligns model behavior with safety expectations
Safety Fine-Tuning
  • Additional training specifically targeting refusal behavior
Output Moderation
  • Screening generated content before it reaches the user
Adversarial Testing
  • Deliberately probing for jailbreak vulnerabilities before attackers do
Behavioral Monitoring
  • Watching for patterns of repeated jailbreak attempts
Human Review Systems
  • Escalating suspicious interactions for manual review
Key Points
  • Jailbreaking mitigation leans heavily on model-level defenses (RLHF, safety fine-tuning), not just input filtering
J4Prompt Injection vs. Jailbreaking

Prompt Injection's objective is to manipulate outputs through embedded commands, altering a specific response. Jailbreaking's objective is to bypass safeguards entirely through roleplay or exploits, producing unsafe, unrestricted behavior more broadly. Injection changes an answer; jailbreaking removes the guardrails.

Key Points
  • Prompt Injection = manipulate a specific output. Jailbreaking = remove safety restrictions broadly.
K Defending Against Prompt Risks K
K1Four Categories of Defense

Defending against prompt risk requires four distinct layers, from organizational policy down to individual human judgment — a technical fix alone is never sufficient.

Organizational Best Practices
  • Establish AI governance policies, audit prompts/outputs regularly, restrict access, monitor abnormal behavior, train employees
Technical Defenses
  • Prompt filtering, output moderation, context segmentation, access control, retrieval isolation, secure logging
Model-Level Defenses
  • RLHF alignment, adversarial training, safety fine-tuning, content moderation layers
Human-Centered Defenses
  • Human-in-the-loop review, ethical oversight boards, escalation mechanisms, incident response plans
Key Points
  • AI security is not only a technical challenge but also a governance, legal, and organizational responsibility
K2Industry-Specific Defense Examples

The same four-layer model applies differently depending on industry — the specific defense implemented reflects that industry's most sensitive data or process.

Healthcare
  • Mask patient identifiers
Finance
  • Restrict transaction prompt access
HR
  • Audit recruitment AI for bias
Government
  • Enforce classified data filtering
Ecommerce
  • Prevent prompt-based fraud generation
Key Points
  • Defense implementation is industry-specific, even when the underlying category is shared
K3Common Indicators of Prompt Abuse

Five behavioral patterns signal a prompt-abuse attempt in progress, useful as a practical monitoring checklist.

Repeated Override Attempts
  • Multiple tries to cancel or bypass system instructions
Requests for Hidden Prompts
  • Directly asking to see system-level instructions
Suspicious Roleplay Scenarios
  • Framing that exists mainly to lower safety guardrails
Malware or Illegal Action Requests
  • Direct or indirect requests for clearly prohibited content
Attempts to Reveal Confidential Data
  • Probing for information the user shouldn't have access to
Key Points
  • These five indicators function as a practical monitoring checklist for abuse detection
L Practical Applications and Exam-Level Insights L
L1Why These Risks Matter in Real Organizations

AI systems are increasingly integrated into banking, healthcare, customer support, government systems, and cybersecurity tools. A single prompt exploit in any of these contexts can leak sensitive data, manipulate outputs, harm customers, and cause compliance violations — the risks covered in this module aren't academic.

Key Points
  • A single prompt exploit can cascade into data leaks, manipulated outputs, customer harm, and compliance violations
L2The Mental Model: Attack Type to Target

The fastest way to distinguish these five risk categories is by asking one question: what does the attack actually target? This mapping resolves nearly every point of confusion between them.

Poisoning
  • Targets Training Data
Prompt Injection
  • Targets Prompt Instructions
Exposure
  • Targets Sensitive Information
Prompt Leaking
  • Targets System Prompts
Jailbreaking
  • Targets Safety Constraints
Key Points
  • Most AI attacks exploit either DATA, PROMPTS, or SAFETY MECHANISMS — identifying which layer is targeted is the module's final, most important insight
L3Common Exam Traps

Three confusions recur most often: confusing Prompt Leaking with Exposure (system instructions vs. personal data), confusing Prompt Injection with Jailbreaking (manipulating one output vs. removing safeguards broadly), and confusing Poisoning with inference-time attacks (training-time corruption vs. runtime manipulation).

Key Points
  • Three recurring exam traps: Leaking vs. Exposure, Injection vs. Jailbreaking, Poisoning vs. inference-time attacks
Module Summary
  • Prompt quality directly determines output quality; effective prompts combine Instructions, Context, Input Data, and an Output Indicator, and can be strengthened with negative prompting.
  • Inference parameters — Temperature, Top P, Top K, Maximum Length, Stop Sequences — let a fixed prompt trade coherence for creativity without changing the wording itself.
  • Zero-Shot, Few-Shot, and Chain-of-Thought are the three foundational prompting techniques, each suited to a different level of task complexity and reasoning demand.
  • Prompt risk splits into five categories — Poisoning (training data), Prompt Injection (runtime prompt), Exposure (sensitive data), Prompt Leaking (system instructions), and Jailbreaking (safety constraints) — each with its own mitigation strategy.
  • Defending against these risks requires four coordinated layers: organizational policy, technical controls, model-level alignment (RLHF, safety fine-tuning), and human review — AI security is a governance responsibility, not just a technical one.

Module Introduction

Modules 1 through 3 covered what AI and ML are, where they're used, and how to build them responsibly. This module answers the practical question underneath all of that: how does a machine learning model actually get built, evaluated, deployed, and kept alive in production? It walks the full lifecycle end to end — using a real Amazon call-routing example — then goes deep on the two halves that determine whether a shipped model is actually any good: rigorous evaluation (datasets, bias-variance, classification and regression metrics, business alignment) and disciplined operations (deployment types, inference strategies, and MLOps).

By the end of this module, the same SageMaker toolchain appears from three different angles: as a development platform (Sections C–H), as an evaluation framework (Sections I–O), and as an operational discipline (Sections P–U) — three views of the same underlying lifecycle from Section A, now with the specific AWS tooling attached to each phase.

A The Machine Learning Development Lifecycle A
A1The Seven Lifecycle Phases

The ML lifecycle spans development, deployment, and maintenance. ML is an iterative loop, not a linear process — the final phase feeds back into the first.

Business Goal Identification
  • Define the business objective before any modeling begins
ML Problem Framing
  • Translate the business goal into a specific ML task
Data Processing
  • Collect, clean, and engineer features from raw data
Model Development
  • Train, tune, and evaluate a predictive model
Model Deployment
  • Put the model into real-world prediction use
Model Monitoring
  • Track performance over time in production
Model Retraining
  • Update the model as new data trends emerge
Key Points
  • Seven phases: Business Goal → Problem Framing → Data Processing → Model Development → Deployment → Monitoring → Retraining
  • ML is an iterative loop, not a linear process — the module's core framing
A2Inside Data Processing

The Data Processing phase itself breaks into three sub-steps, each building on the last.

Data Collection
  • Gathering raw data from source systems
Data Preprocessing
  • Cleaning and transforming that raw data
Feature Engineering
  • Extracting the variables actually useful for prediction
Key Points
  • Data Processing sub-steps: Data Collection, Data Preprocessing, Feature Engineering
B Real-World ML Example: Amazon Call Center B
B1Problem Context and ML Conversion

The business problem was inefficient call routing, causing delays, added cost, and poor customer experience. Business problem → ML problem transformation: the goal became predicting the correct agent skill needed for a call — framed specifically as multiclass classification, since a call can route to one of several possible skill categories.

Key Points
  • ML problem type: Multiclass Classification — predict the correct agent skill
B2The Data Phase

Historical customer call data fed the model, using supervised learning with three key features.

Recent Orders
  • What the customer has recently purchased
Device Ownership
  • Whether the customer owns a Kindle, for example
Membership Status
  • The customer's account/membership tier
Key Points
  • Learning type: Supervised Learning, using historical customer call data
B3Data Preparation and the Train/Validation/Test Split

Preprocessing included data cleaning (removing errors) and label refinement — combining similar labels, such as merging multiple distinct “Kindle issue” labels into one. Visualization helped the team understand call-type distributions before training began.

Data was split into three sets: Train (70–80%), Validation (10–15%), and Test (10–15%). Never evaluate using training data — doing so measures memorization, not generalization.

Key Points
  • Data split: Train 70–80%, Validation 10–15%, Test 10–15%
  • Never evaluate using training data — the module's core caution
B4Optimization and Deployment Outcome

Model optimization combined hyperparameter tuning (adjusting learning behavior) with further feature engineering (improving input representation). The deployment outcome was measurable: fewer call transfers and an improved customer experience. Success metric = business improvement, not a raw accuracy number in isolation.

Key Points
  • Success metric = business improvement — the module's core framing, echoed later in Section O

K Model Fit: Underfitting vs. Overfitting K
K1Three Fit States

Model Fit describes the relationship between prediction and actual values. Balance = generalization performance — the goal state below.

Overfitting
  • Performs well on training data, poorly on unseen data — caused by memorizing noise
Underfitting
  • Performs poorly even on training data — caused by a model that's too simple
Balanced Model
  • Performs well on both training and unseen data — the goal: optimal generalization
Key Points
  • Balance = generalization performance — the module's core insight for this section
L The Bias-Variance Trade-Off L
L1Bias vs. Variance

Bias is error from wrong assumptions; Variance is sensitivity to data fluctuations. High Bias → underfitting; High Variance → overfitting. The ideal model has low bias and low variance together.

The module's bullseye analogy: the bullseye center represents the true value, spread represents variance, and distance from center represents bias. The trade-off is to minimize both bias and variance at once, not to trade one for the other.

Key Points
  • High Bias → underfitting. High Variance → overfitting.
  • Bullseye analogy: center = true value, spread = variance, distance from center = bias
M Classification Metrics M
M1The Confusion Matrix

The confusion matrix is the foundation of classification metrics — every metric in this section is computed from these four values.

True Positive (TP)
  • A correct positive prediction
True Negative (TN)
  • A correct negative prediction
False Positive (FP)
  • An incorrect positive prediction
False Negative (FN)
  • A missed positive case
Key Points
  • Confusion matrix = foundation of classification metrics — the module's core framing
M2Accuracy, Precision, and Recall

Three metrics answer three different questions about classifier quality, and matter most in different situations.

Accuracy
  • (TP+TN)/(TP+TN+FP+FN) — misleading with imbalanced data
Precision
  • TP/(TP+FP) — use when false positives are costly, e.g. spam filtering
Recall (Sensitivity)
  • TP/(TP+FN) — use when false negatives are critical, e.g. disease detection
Key Points
  • Precision vs Recall = an error-cost trade-off — the module's core insight for this subsection
M3F1 Score and AUC-ROC

F1 Score balances precision and recall, used when both false positives and false negatives matter. AUC-ROC plots the ROC curve (true positive rate vs. false positive rate) and produces an AUC separability score: roughly 0.5 means random guessing, and higher is better. AUC-ROC is a threshold-independent performance metric — it shows performance across every possible decision threshold at once.

Key Points
  • AUC-ROC is threshold-independent — it shows performance across thresholds, not at one fixed cutoff
N Regression Metrics N
N1MSE and R²

Mean Squared Error (MSE) averages the squared difference between prediction and actual value — lower MSE means a better model. R Squared (R²) measures the proportion of variance explained by the model, ranging 0 to 1, where closer to 1 means a better fit. MSE and R² complement each other: MSE measures error magnitude, R² measures goodness of fit.

Key Points
  • MSE = error magnitude (lower is better). R² = goodness of fit (closer to 1 is better). They complement each other.
O Business Metrics and Model Evaluation O
O1Business Metrics and Alignment

Business metrics measure real-world impact — increasing sales, reducing cost, improving retention. ML metrics must reflect business tolerance for errors and their cost implications: a false positive might mean increased operational cost, while a false negative might mean a missed opportunity.

Key Points
  • Business metric examples: increase sales, reduce cost, improve retention
O2Custom Metrics and Experimentation

Two experimentation techniques and a custom-metric approach let a team validate that a model actually improves the business, not just its technical score.

Cost Function
  • Assigns explicit value to predictions and errors, optimizing directly for business impact
A/B Testing
  • Compares model versions head-to-head with real traffic
Canary Deployment
  • Gradually rolls out a new model to limit blast radius if something goes wrong
Key Points
  • Best model = one that optimizes business KPIs — the module's closing insight for this section
P Model Deployment P
P1Self-Hosted vs. Managed API

Self-Hosted API deploys on your own infrastructure (on-premise or cloud VMs/containers) — full control and customization, but high operational overhead. Managed API (e.g. Amazon SageMaker AI) removes infrastructure management for faster deployment and built-in scalability, at the cost of less control. The choice is control versus convenience.

Key Points
  • Deployment choice = control vs. convenience trade-off — the module's core framing
P2SageMaker Deployment Capabilities

SageMaker's managed deployment offers four specific capabilities that remove common operational burden.

One-Click or API Deployment
  • Deploy without manual infrastructure setup
Automatic Scaling
  • Capacity adjusts to traffic automatically
Secure HTTPS Endpoints
  • Production-grade, encrypted access by default
Multi-Model Hosting
  • Serve multiple models from shared infrastructure
Key Points
  • Four SageMaker deployment capabilities: One-Click/API Deployment, Automatic Scaling, Secure HTTPS Endpoints, Multi-Model Hosting
P3Four Inference Types

Inference type depends on latency + workload — choosing the wrong one wastes cost or fails the use case's latency requirement.

Real-Time Inference
  • Low latency, for interactive applications
Batch Transform
  • Large dataset processing, no persistent endpoint
Asynchronous Inference
  • Handles large payloads and long processing time via queued requests
Serverless Inference
  • No infrastructure management, scales automatically, best for intermittent traffic
Key Points
  • Inference type depends on latency + workload — the module's core selection rule for this section
Q Fundamental Concepts of MLOps Q
Q1What MLOps Is

MLOps operationalizes the ML lifecycle, combining people, processes, and technology, spanning development through monitoring. MLOps = DevOps for Machine Learning — but it isn't just DevOps renamed: ML models degrade over time as data changes, requiring continuous monitoring that traditional software doesn't need in the same way.

Key Points
  • MLOps = DevOps for Machine Learning — but with ML-specific degradation and monitoring needs
Q2Benefits of MLOps

MLOps ensures models stay useful over time — delivered through four measurable benefits.

Productivity
  • Faster experimentation, access to curated datasets
Reliability
  • CI/CD improves consistency across releases
Repeatability
  • Automated, reproducible pipelines
Auditability
  • Tracking data, models, and experiments; monitoring bias and drift
Key Points
  • MLOps ensures models stay useful over time — the module's key insight for this section
  • MLOps goals: faster ML lifecycle, improved model quality, strong collaboration, transparency and governance
R Key Principles of MLOps R
R1Version Control and Automation

Version Control tracks code, data, and models together, enabling rollback and reproducibility. Automation covers data ingestion, training, testing, and deployment — removing manual steps that would otherwise block reliable, repeatable releases.

Key Points
  • Version Control tracks code + data + models. Automation covers ingestion, training, testing, deployment.
R2CI/CD for ML

CI/CD in ML includes training and monitoring — two stages beyond what traditional software CI/CD covers.

Continuous Integration
  • Tests code, data, and models together
Continuous Delivery
  • Deploys models automatically
Continuous Training
  • Retrains models as new data arrives
Continuous Monitoring
  • Tracks both model and business metrics after deployment
Key Points
  • CI/CD in ML includes training + monitoring — the module's key insight for this subsection
R3Model Governance

Model Governance ensures fairness, compliance, and security through documentation, validation processes, and stakeholder collaboration. Governance = trust + compliance in ML systems — the same theme covered in depth in Module 3 and Module 7, applied here specifically to the MLOps pipeline.

Key Points
  • Governance = trust + compliance in ML systems — the module's key insight for this subsection

T MLOps Implementation Pipeline T
T1Training Pipeline, Registration, and Approval

The Training Pipeline covers data preparation, model building, and model evaluation. A model is registered in the Model Registry only if it meets performance criteria, and then goes through Model Approval — explicitly approved or rejected before it can move forward.

The Deployment Pipeline follows a fixed sequence: Source → Build → Deploy → Test → Promote. Once live, Production Monitoring tracks infrastructure, data quality, and model performance continuously. Pipelines enable automated ML systems — removing manual handoffs between each stage.

Key Points
  • Deployment pipeline sequence: Source → Build → Deploy → Test → Promote
  • Pipelines enable automated ML systems — the module's key insight for this section

Module Summary
  • The ML lifecycle is a seven-phase iterative loop — Business Goal, Problem Framing, Data Processing, Model Development, Deployment, Monitoring, Retraining — not a one-time linear process.
  • Amazon SageMaker AI is the full end-to-end ML lifecycle platform, with dedicated tools for data preparation (Data Wrangler), feature management (Feature Store), training (Training Jobs, Canvas, JumpStart), evaluation (Experiments), deployment, and monitoring (Model Monitor).
  • Evaluation exists to measure generalization, not memorization: the train/validation/test split, the underfitting/overfitting/balanced fit spectrum, and the bias-variance trade-off all serve that one goal.
  • Classification metrics (Accuracy, Precision, Recall, F1, AUC-ROC) and regression metrics (MSE, R²) each answer a different question about model quality — and the best model is ultimately the one that optimizes business KPIs, not just technical scores.
  • Deployment splits into self-hosted vs. managed, with four inference types (real-time, batch, async, serverless) matched to workload; MLOps — versioning, automation, CI/CD extended with continuous training and monitoring, and governance — is what keeps a deployed model reliable over time.

Module Introduction

Full lifecycle for shipping a Gen AI application: define the use case, select a Foundation Model, improve it with prompting/RAG/fine-tuning, evaluate, then deploy — looping back as feedback arrives.

A The Machine Learning Development Lifecycle A
A1The Seven Lifecycle Phases

The ML lifecycle spans development, deployment, and maintenance.

Business Goal Identification
  • Define the business objective before any modeling begins
ML Problem Framing
  • Translate the business goal into a specific ML task
Data Processing
  • Collect, clean, and engineer features from raw data
Model Development
  • Train, tune, and evaluate a predictive model
Model Deployment
  • Put the model into real-world prediction use
Model Monitoring
  • Track performance over time in production
Model Retraining
  • Update the model as new data trends emerge
KeySeven phases: Business Goal → Problem Framing → Data Processing → Model Development → Deployment → Monitoring → Retraining›ML is an iterative loop, not a linear process — the module's core framing
A2Inside Data Processing

The Data Processing phase itself breaks into three sub-steps, each building on the last.

Data Collection
  • Gathering raw data from source systems
Data Preprocessing
  • Cleaning and transforming that raw data
Feature Engineering
  • Extracting the variables actually useful for prediction
KeyData Processing sub-steps: Data Collection, Data Preprocessing, Feature Engineering
B Real-World ML Example: Amazon Call Center B
B1Problem Context and ML Conversion

The business problem was inefficient call routing, causing delays, added cost, and poor customer experience.

KeyML problem type: Multiclass Classification — predict the correct agent skill
B2The Data Phase

Historical customer call data fed the model, using supervised learning with three key features.

Recent Orders
  • What the customer has recently purchased
Device Ownership
  • Whether the customer owns a Kindle, for example
Membership Status
  • The customer's account/membership tier
KeyLearning type: Supervised Learning, using historical customer call data
B3Data Preparation and the Train/Validation/Test Split

Preprocessing included data cleaning (removing errors) and label refinement — combining similar labels, such as merging multiple distinct “Kindle issue” labels into one. Data was split into three sets: Train (70–80%), Validation (10–15%), and Test (10–15%).

KeyData split: Train 70–80%, Validation 10–15%, Test 10–15%›Never evaluate using training data — the module's core caution
B4Optimization and Deployment Outcome

Model optimization combined hyperparameter tuning (adjusting learning behavior) with further feature engineering (improving input representation).

KeySuccess metric = business improvement — the module's core framing, echoed later in Section O

I Machine Learning Model Performance Evaluation I
I1What Evaluation Measures

Model Evaluation assesses predictive performance, with the goal of ensuring the model generalizes to unseen data.

KeyEvaluation = generalization, not memorization — the module's central thesis for Sections I–O
J Evaluation Datasets J
J1Training, Validation, and Test Sets

Three datasets serve three distinct purposes across the evaluation process — using one for another's purpose invalidates the result.

Training Set
  • Used to actually train the model
Validation Set
  • Used for hyperparameter tuning and iterative model improvement
Test Set
  • Final, unbiased evaluation before deployment
KeyTest set must remain untouched until final evaluation — the module's core rule
K Model Fit: Underfitting vs. Overfitting K
K1Three Fit States

Model Fit describes the relationship between prediction and actual values.

Overfitting
  • Performs well on training data, poorly on unseen data — caused by memorizing noise
Underfitting
  • Performs poorly even on training data — caused by a model that's too simple
Balanced Model
  • Performs well on both training and unseen data — the goal: optimal generalization
KeyBalance = generalization performance — the module's core insight for this section
L The Bias-Variance Trade-Off L
L1Bias vs. Variance

Bias is error from wrong assumptions; Variance is sensitivity to data fluctuations. The module's bullseye analogy: the bullseye center represents the true value, spread represents variance, and distance from center represents bias.

KeyHigh Bias → underfitting. High Variance → overfitting.›Bullseye analogy: center = true value, spread = variance, distance from center = bias
M Classification Metrics M
M1The Confusion Matrix

The confusion matrix is the foundation of classification metrics — every metric in this section is computed from these four values.

True Positive (TP)
  • A correct positive prediction
True Negative (TN)
  • A correct negative prediction
False Positive (FP)
  • An incorrect positive prediction
False Negative (FN)
  • A missed positive case
KeyConfusion matrix = foundation of classification metrics — the module's core framing
M2Accuracy, Precision, and Recall

Three metrics answer three different questions about classifier quality, and matter most in different situations.

Accuracy
  • (TP+TN)/(TP+TN+FP+FN) — misleading with imbalanced data
Precision
  • TP/(TP+FP) — use when false positives are costly, e.g. spam filtering
Recall (Sensitivity)
  • TP/(TP+FN) — use when false negatives are critical, e.g. disease detection
KeyPrecision vs Recall = an error-cost trade-off — the module's core insight for this subsection
M3F1 Score and AUC-ROC

F1 Score balances precision and recall, used when both false positives and false negatives matter.

KeyAUC-ROC is threshold-independent — it shows performance across thresholds, not at one fixed cutoff
N Regression Metrics N
N1MSE and R²

Mean Squared Error (MSE) averages the squared difference between prediction and actual value — lower MSE means a better model.

KeyMSE = error magnitude (lower is better). R² = goodness of fit (closer to 1 is better). They complement each other.

Two experimentation techniques and a custom-metric approach let a team validate that a model actually improves the business, not just its technical score.

P3Four Inference Types

Inference type depends on latency + workload — choosing the wrong one wastes cost or fails the use case's latency requirement.

Real-Time Inference
  • Low latency, for interactive applications
Batch Transform
  • Large dataset processing, no persistent endpoint
Asynchronous Inference
  • Handles large payloads and long processing time via queued requests
Serverless Inference
  • No infrastructure management, scales automatically, best for intermittent traffic
KeyInference type depends on latency + workload — the module's core selection rule for this section

Module Summary
  • The ML lifecycle is a seven-phase iterative loop — Business Goal, Problem Framing, Data Processing, Model Development, Deployment, Monitoring, Retraining — not a one-time linear process.
  • Amazon SageMaker AI is the full end-to-end ML lifecycle platform, with dedicated tools for data preparation (Data Wrangler), feature management (Feature Store), training (Training Jobs, Canvas, JumpStart), evaluation (Experiments), deployment, and monitoring (Model Monitor).
  • Evaluation exists to measure generalization, not memorization: the train/validation/test split, the underfitting/overfitting/balanced fit spectrum, and the bias-variance trade-off all serve that one goal.
  • Classification metrics (Accuracy, Precision, Recall, F1, AUC-ROC) and regression metrics (MSE, R²) each answer a different question about model quality — and the best model is ultimately the one that optimizes business KPIs, not just technical scores.
  • Deployment splits into self-hosted vs. managed, with four inference types (real-time, batch, async, serverless) matched to workload; MLOps — versioning, automation, CI/CD extended with continuous training and monitoring, and governance — is what keeps a deployed model reliable over time.

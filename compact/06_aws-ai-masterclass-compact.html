Module Introduction

Full lifecycle for shipping a Gen AI application: define the use case, select a Foundation Model, improve it with prompting/RAG/fine-tuning, evaluate, then deploy — looping back as feedback arrives.

A Foundations of FM Optimization A
A1What Optimization Means

Foundation Model Optimization is the practice of improving a deployed model's output quality and relevance after it has already been selected. The goal is not abstract quality — it is aligning model outputs with business objectives.

KeyFM Optimization = improving output quality and relevance post-selection›The target is business alignment, not abstract benchmark performance
A2Core Objectives

Optimization work in this module organizes around four core objectives, each covered in its own section below.

Embedding Storage
  • Store vector representations of data in a database for fast retrieval
Agents
  • Automate multi-step workflows that connect the model to backend systems
Evaluation
  • Measure how effectively the FM performs against business goals
Fine-Tuning
  • Adapt the model's weights to a specific domain or task
KeyFour core objectives: Embedding Storage, Agents, Evaluation, Fine-Tuning›Best performance = Data + Context + Iteration — the module's key insight
B Business Case: AnyCompany Telecom Chatbot B
B1Problem & Goals

AnyCompany, a telecom provider, faces high support cost and operational inefficiency from its customer service line. [+] Notice these are exactly the kind of concrete, measurable success criteria that Module 5's Use Case Structure calls for — a percentage reduction and a satisfaction floor, not a vague “improve support.”.

KeyGoal: reduce tickets by 70%, raise satisfaction to ≥ 4/5
B2ML Challenges

Off-the-shelf LLMs lack domain-specific knowledge — they don't know AnyCompany's plans, policies, or account systems.

KeyGeneric models lack domain knowledge by default›Two needs: enterprise data integration + task automation capability
B3Solution Components

AnyCompany's solution combines three components, each covered in depth later in this module.

Knowledge Base

Built from chat logs, support tickets, and call recordings.

Data Preparation

Collection, cleaning, and anonymization of that raw material before it's usable.

Automation

Agents that can modify user plans and execute backend actions, turning a conversation into a completed task..

KeyGeneric models need context to be useful — this is the module's core diagnostic›Solution = Knowledge Base + Data Preparation + Automation
C Retrieval-Augmented Generation (RAG) C
C1RAG and Enterprise Data Integration

RAG combines retrieval with generation to improve accuracy using external data. RAG = context injection without retraining: the model's weights never change, only what it's given to read before answering.

KeyRAG's enterprise data sources: documents, reports, manuals›RAG = context injection without retraining — the module's core definition
C2Vector Embeddings

An embedding is a numerical representation of data — text, in this case — that captures its semantic meaning as a vector, enabling mathematical comparison between pieces of text. The key insight: similar words produce similar vectors.

KeyEmbedding = numerical vector representation capturing semantic meaning›Similar meaning → similar vectors (Sea ≈ Ocean; Stapler ≠ Ocean)
C3Vector Databases & Similarity Search

A vector database stores embeddings and enables fast similarity search, retrieving the most relevant context for a given query.

KeyVector DB purpose: store embeddings, enable fast similarity search, retrieve relevant context›Two search techniques: k-NN (nearest vector search), Cosine Similarity (angle-based)
C4AWS Vector Database Options

AWS offers several managed options for storing and querying vector embeddings, each fitting a different existing infrastructure.

Amazon OpenSearch
  • Available in Provisioned and Serverless modes for vector search
pgvector
  • A PostgreSQL extension available on Amazon RDS and Aurora
Amazon Kendra
  • An enterprise search service usable as a managed retrieval layer
KeyThree AWS vector DB options: Amazon OpenSearch, pgvector (RDS/Aurora), Amazon Kendra
D Agents in Generative AI Systems D
D1Key Functions of Agents

An agent's role is to act as an intermediary between the model and the outside world.

Intermediary Operations

Connects the model to backend systems, translating a model's decision into a system-level call..

Action Execution

Actually performs actions.

Feedback Integration

Feeds outcomes of those actions back into the system, improving the model's behavior over time..

KeyAgents enable real-world actions from AI outputs — not just text generation›Three functions: Intermediary Operations, Action Execution, Feedback Integration
D2AnyCompany Agent Example

AnyCompany's telecom chatbot deploys three distinct agents, each with a narrow, well-defined responsibility — a pattern worth copying rather than building one do-everything agent.

Agent 1
  • Manages customer plans — upgrades, downgrades, modifications
Agent 2
  • Updates the knowledge base as new information becomes available
Agent 3
  • Collects feedback from completed interactions
KeyAnyCompany's three agents: plan management, knowledge base updates, feedback collection
E Evaluating Generative AI Systems E
E1Evaluation Methods

Evaluating a Gen AI system means asking whether it's actually effective for the business, not just whether it runs.

Human Evaluation

Focuses on user experience, relevance, and creativity.

Benchmark Datasets

Provide quantitative evaluation across accuracy, speed, and scalability.

LLM-as-Judge

Uses another model to compare a generated answer against a reference answer, scoring accuracy, relevance, and completeness automatically..

KeyBest evaluation = Human + Metrics, combined rather than either alone›LLM-as-Judge automates evaluation using another model as the grader
E2Benchmark Creation Steps

Building a benchmark dataset follows a fixed three-step sequence: first, Questions are created by subject-matter experts (SMEs); second, Context — the relevant supporting information — is identified for each question; third, Answers are defined as the ground truth the model's output will be compared against.

KeyBenchmark creation order: Questions (by SMEs) → Context → Answers (ground truth)

G Fine-Tuning Foundation Models G
G1Why Fine-Tune: Benefits

Fine-tuning adapts a model to a specific domain by continuing training on domain data.

Increased Specificity
  • Outputs speak the domain's actual vocabulary and conventions
Improved Accuracy
  • Fewer errors on domain-specific tasks versus a generic model
Reduced Bias
  • Domain-curated data can correct skew present in general training data
Enhanced Efficiency
  • Shorter prompts needed since domain knowledge is baked into the weights
KeyFine-tuning = specialization of general models›Four benefits: Increased Specificity, Improved Accuracy, Reduced Bias, Enhanced Efficiency
G2Fine-Tuning Approaches

Five distinct approaches fall under the fine-tuning umbrella, differing in what data they use and how they change the model.

Instruction Tuning
  • Trains the model to follow structured input-output examples
RLHF
  • Optimizes model behavior using human feedback on output quality
Domain Adaptation
  • Uses specialized, domain-specific datasets for retraining
Transfer Learning
  • Reuses knowledge already captured in a pretrained model as a starting point
Continuous Pretraining
  • Updates a model's knowledge with new data over time, without a full retrain
KeyFive approaches: Instruction Tuning, RLHF, Domain Adaptation, Transfer Learning, Continuous Pretraining
H Data Preparation for Fine-Tuning H
H1Initial Training Data vs. Fine-Tuning Data

A Foundation Model's initial training data is large-scale, diverse, and general-purpose — built to give the model broad competence. [+] This is why fine-tuning datasets are usually orders of magnitude smaller than pretraining datasets — the goal shifts from breadth to precision.

KeyInitial training data: large-scale, diverse, general-purpose›Fine-tuning data: specific, relevant, high-quality — small and curated, not broad
H2Key Data Preparation Steps

Five steps determine whether fine-tuning data is actually usable, and the module is explicit that quality outweighs quantity throughout this process.

Data Curation
  • Selecting and assembling the dataset from raw sources
Labeling
  • Attaching correct target outputs or annotations to each example
Governance & Compliance
  • Ensuring data use meets legal and regulatory requirements
Bias Checking
  • Identifying and correcting skew that would propagate into the fine-tuned model
Feedback Integration
  • Incorporating real usage feedback back into future data preparation cycles
KeyQuality > Quantity in fine-tuning data — the section's key insight›Five steps: Data Curation, Labeling, Governance & Compliance, Bias Checking, Feedback Integration
I Model Evaluation Metrics I
I1ROUGE, BLEU, and BERTScore

The same three metrics from Module 5 reappear here with sharper definitions of what each one actually measures — the differences matter when choosing which to report.

ROUGE
  • Measures overlap with a reference text
BLEU
  • Measures precision of N-grams against a reference
BERTScore
  • Measures semantic similarity using contextual embeddings
KeyROUGE = completeness (recall) · BLEU = accuracy of phrasing (precision + brevity penalty) · BERTScore = meaning (semantic similarity)›Combine metrics for full evaluation — no single one tells the whole story
Module Summary
  • FM Optimization means aligning a selected model's outputs with real business objectives, organized around four objectives: Embedding Storage, Agents, Evaluation, and Fine-Tuning.
  • RAG injects enterprise context without retraining — vector embeddings capture semantic meaning, and vector databases (Amazon OpenSearch, pgvector, Amazon Kendra) make similarity search fast via k-NN or Cosine Similarity.
  • Agents turn model output into real-world action, serving as the intermediary, executor, and feedback channel between the model and backend systems.
  • Strong evaluation combines Human Evaluation, Benchmark Datasets, and LLM-as-Judge — no single method is sufficient alone.
  • Fine-tuning specializes a general model through five approaches (Instruction Tuning, RLHF, Domain Adaptation, Transfer Learning, Continuous Pretraining), but only pays off when the fine-tuning data is curated for quality over quantity.

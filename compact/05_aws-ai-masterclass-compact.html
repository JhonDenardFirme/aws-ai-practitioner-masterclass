Module Introduction

Full lifecycle for shipping a Gen AI application: define the use case, select a Foundation Model, improve it with prompting/RAG/fine-tuning, evaluate, then deploy — looping back as feedback arrives.

A The Generative AI Application Lifecycle A
A1Lifecycle Phases

Every generative AI application, regardless of use case, moves through the same six-phase lifecycle. The phases run in order the first time through: Define the business use case, Select a Foundation Model (FM), Improve its performance for the task, Evaluate the results against real criteria, and Deploy into production.

KeySix phases: Define → Select FM → Improve → Evaluate → Deploy → Iterate›The lifecycle is iterative — teams cycle back to earlier phases as feedback arrives, not just run it once
A2Iteration as a Core Property

Nondeterminism (the tendency of a generative model to produce different outputs for the same input) and evolving user needs mean a generative AI system is never truly “finished.” Iteration loops feedback from evaluation and real usage back into use-case refinement, prompt design, or model selection. [+] In AWS terms, this iteration typically shows up as repeated passes through Amazon Bedrock experimentation, prompt versioning, and re-evaluation — rather than a one-time model swap.

KeyGen AI Lifecycle is iterative — continuously refined, never a single pass›Iteration is triggered by evaluation results, user feedback, and changing requirements
A3Capabilities of Generative AI

Generative AI's practical value comes from seven core capabilities.

Adaptability
  • Works across multiple domains and task types without redesign
Responsiveness
  • Generates outputs in real time, supporting interactive use cases
Simplicity
  • Automates workflows that previously required multiple manual steps
Creativity
  • Produces novel ideas, phrasing, or content rather than retrieving fixed answers
Data Efficiency
  • Learns generalizable patterns from large-scale training data
Personalization
  • Tailors outputs to an individual user's context or preferences
Scalability
  • Generates content at a volume no human team could match
KeySeven capabilities: Adaptability, Responsiveness, Simplicity, Creativity, Data Efficiency, Personalization, Scalability
A4Challenges of Generative AI

The same properties that make generative AI powerful also introduce risk.

Regulatory Violations
  • Misuse of sensitive, restricted, or regulated data in training or inference
Social Risks
  • Outputs that are harmful, misleading, or reinforce bias at scale
Privacy Concerns
  • Exposure of personal or confidential information through generated output
Toxicity
  • Offensive, unsafe, or inappropriate generated content
Hallucinations
  • Plausible-sounding but factually incorrect output presented with confidence
Interpretability
  • Difficulty explaining why a model produced a specific output
Nondeterminism
  • Inconsistent output for the same input across separate runs
KeyHallucinations + nondeterminism are the two most cited risks in production Gen AI systems›Every challenge listed here maps to a mitigation covered later in this module (RAG for hallucination, evaluation for interpretability, etc.)
B Defining a Use Case B
B1Why Use Case Definition Comes First

A generative AI project that begins with model selection instead of use-case definition tends to produce technically impressive systems that solve the wrong problem. [+] A useful diagnostic: if a team cannot state the use case in one sentence with a measurable outcome, the project is not ready to move to Foundation Model selection.

KeyA bad use case definition produces a failed AI project, regardless of model quality›Use case definition also determines the success metrics used later in Evaluation
B2Business Use Case Structure

A well-formed use case document follows a standard structure.

Use Case Name
  • A short, descriptive identifier for the use case
Brief Description
  • A one-paragraph summary of the objective
Actors
  • The users or systems that interact with the solution
Preconditions
  • Conditions that must be true before the flow can start
Basic Flow
  • The main, step-by-step success path
Alternative Flows
  • Error handling and edge-case paths
Postconditions
  • The system's final state after successful execution
Business Rules
  • Policies and constraints the solution must respect
Nonfunctional Requirements
  • Performance, security, and usability requirements
Assumptions
  • Conditions expected to hold true in the environment
Notes
  • Additional clarifications that don't fit elsewhere
Key11 standard fields — Name, Description, Actors, Preconditions, Basic Flow, Alternative Flows, Postconditions, Business Rules, NFRs, Assumptions, Notes›Alternative Flows are what most teams skip — and what causes production incidents later
B3Key Metrics for Generative AI Use Cases

The use case document defines not just what the system does, but how success will be measured.

Cost Savings
  • Reduction in operational expense versus the current process
Time Savings
  • Faster completion of the target task
Quality Improvement
  • Measurably better outputs than the current baseline
Customer Satisfaction
  • Improved experience, typically tracked via survey or NPS
Productivity Gains
  • Increased throughput or efficiency per team member
KeyThe use case defines both metrics and success criteria — evaluation later in the lifecycle measures against these, not against generic benchmarks alone
B4Common Approaches

Most generative AI use cases fall into one of five common approach patterns.

Process Automation
  • Automating repetitive, rules-based tasks
Decision Support
  • Assisting — not replacing — human decision-making
Personalization
  • Tailoring experiences to individual users
Content Generation
  • Creating text, images, or other media
Innovation Exploration
  • Generating new ideas or design directions
KeyFive approach patterns: Process Automation, Decision Support, Personalization, Content Generation, Innovation Exploration
C Selecting a Foundation Model C
C1What a Foundation Model Is

A Foundation Model (FM) is a pre-trained, large-scale model that serves as the base for a generative AI application. The central insight of FM selection is that there is no universally “best” model.

KeyAn FM is the pre-trained base every Gen AI application builds on›No single FM is universally best — selection is a trade-off exercise, not a leaderboard lookup
C2Key Selection Criteria

Choosing an FM means weighing ten criteria against the use case's actual constraints.

Cost
  • Compute expense, licensing fees, and customization cost
Modality
  • Text, image, audio, or multimodal input/output support
Latency
  • How fast the model must respond for the use case
Multilingual Support
  • Which languages the model handles well
Model Size
  • Trade-off between performance and resource consumption
Model Complexity
  • Capability versus difficulty of deployment and operation
Customization Ability
  • Whether and how well the model supports fine-tuning
Input/Output Length
  • Context window and generation length limitations
Responsibility
  • Bias, safety, and ethical behavior of the model
Deployment Compatibility
  • Ease of integration into the target production environment
Key10 criteria — Cost, Modality, Latency, Multilingual Support, Size, Complexity, Customization, I/O Length, Responsibility, Deployment Compatibility›Model selection = trade-off between cost, performance, and complexity, not a single winning score
C3Example Models on Amazon Bedrock

Amazon Bedrock provides managed access to several Foundation Model families, each with a different strength profile.

AI21 Jurassic-2

Optimized for text generation and summarization tasks where fluent, on-topic prose matters most..

Amazon Titan

AWS's own model family, offering embeddings alongside text and image generation.

Anthropic Claude

Strong at reasoning, multi-turn chat, and coding tasks, making it a common choice for assistant-style and developer-facing applications..

Cohere Command

Tuned for a broad range of natural language processing (NLP) tasks such as classification and extraction..

Meta Llama

A general-purpose large language model (LLM) family suited to a wide range of tasks without heavy specialization..

Mistral Large

Positioned for high reasoning capability, competing with larger models at a smaller footprint..

KeyBedrock offers multiple FM families — AI21 Jurassic-2, Amazon Titan, Anthropic Claude, Cohere Command, Meta Llama, Mistral Large — each with a distinct strength profile›Picking a model family starts from the task profile, not brand familiarity
D Improving Foundation Model Performance D
D1Prompt Engineering: Core Techniques

Prompt Engineering is the design of inputs to control a model's output — and it is the primary control mechanism available in generative AI, since it requires no retraining and takes effect immediately.

Design
  • Writing clear, structured prompts with explicit instructions
Augmentation
  • Adding examples or constraints to narrow the model's behavior
Tuning
  • Iteratively refining a prompt based on observed output quality
Ensembling
  • Combining multiple prompts or prompt variants for a stronger result
Mining
  • Systematically discovering prompt phrasings that perform well
KeyPrompt engineering is the primary control mechanism in Gen AI — no retraining required›Five techniques: Design, Augmentation, Tuning, Ensembling, Mining
D2Common Prompting Methods

Beyond general technique, several named prompting patterns recur across production systems.

Zero-shot
  • Asking the model to perform a task with no example provided
Few-shot
  • Providing a small number of examples inside the prompt to guide format and style
Chain-of-thought (CoT)
  • Prompting the model to reason step-by-step before answering
Tree-of-thought (ToT)
  • Exploring multiple reasoning branches before selecting the best path
Self-consistency
  • Sampling multiple reasoning paths and taking the majority answer
ReAct
  • Interleaving reasoning steps with tool-use actions
KeySix named methods: Zero-shot, Few-shot, CoT, ToT, Self-consistency, ReAct›CoT and ToT both improve reasoning quality; ToT explores more paths at higher cost
D3Retrieval Augmented Generation (RAG)

RAG combines a retrieval system with a generative model: the retrieval system fetches relevant documents or knowledge, and the generative model synthesizes a response using that retrieved data instead of relying solely on what it memorized during training. RAG reduces hallucinations by grounding generation in real, current data — which is why it is the default technique most production systems reach for before considering fine-tuning.

KeyRAG = Retrieval System + Generative Model›RAG directly targets the hallucination challenge from Section A by grounding answers in retrieved facts
D4RAG Business Applications

RAG's grounding property makes it especially valuable in three recurring business application patterns.

Question-Answering Systems
  • Answering user questions using an organization's own documents
Knowledge Base Enhancement
  • Surfacing accurate information from large internal knowledge bases
Content Generation
  • Producing content that must stay factually grounded in source material
KeyRAG's three core applications: Q&A systems, knowledge base enhancement, grounded content generation
D5Fine-Tuning

Fine-tuning retrains a pre-trained model on domain-specific data, improving task-specific performance beyond what prompting alone can achieve. The fine-tuning process follows a fixed sequence: start with a pre-trained model, prepare a domain dataset, add task-specific layers, train on the domain data, then evaluate and iterate.

KeyFine-tuning methods: Instruction Tuning, RLHF›Process: pre-trained model → prepare dataset → add task-specific layers → train → evaluate & iterate
D6Model from Scratch & the Trade-Off Summary

Building a model from scratch means training entirely new weights on original data — the highest-cost, highest-customization option. Most real systems combine RAG with a pre-trained model rather than reaching for fine-tuning or from-scratch training, since it delivers the largest accuracy gain for the lowest cost.

Pre-trained
  • Lowest cost
Fine-tuned
  • Balanced cost and control
From Scratch
  • Highest cost
KeyTrade-off spectrum: Pre-trained (cheap, low control) → Fine-tuned (balanced) → From Scratch (expensive, full control)›Most production systems land on RAG + pre-trained, not fine-tuning or from-scratch
D7Agents for Automation

An agent is an autonomous task executor that manages multi-step workflows — chaining reasoning, tool calls, and decisions without a human directing each step.

Task Coordination
  • Sequencing and managing multi-step workflows
Logging and Reporting
  • Recording actions taken and outcomes produced
Scalability
  • Handling many concurrent workflow instances
System Integration
  • Connecting to external tools, APIs, and data sources
KeyAgents enable complex, multi-step automation beyond a single prompt-response cycle›Four core functions: Task Coordination, Logging/Reporting, Scalability, System Integration
E Evaluating Foundation Models E
E1Evaluation Methods

No single evaluation method captures everything that matters.

Human Evaluation

Qualitative judgment from real reviewers.

Benchmark Datasets

Standardized datasets that allow apples-to-apples comparison between different models or configurations..

Automated Metrics

Fast, scalable scoring that can run continuously, but is not a substitute for human judgment on nuanced quality..

KeyAutomated metrics ≠ human judgment — they measure different things and both are needed
E2Benchmark Examples

[+] Four benchmark datasets recur constantly in FM evaluation literature and AWS documentation.

GLUE
  • General Language Understanding Evaluation — a suite of core NLP understanding tasks
SuperGLUE
  • A harder successor to GLUE, designed as earlier models saturated it
SQuAD
  • Stanford Question Answering Dataset — benchmarks reading-comprehension QA
WMT
  • Workshop on Machine Translation — benchmarks translation quality
KeyFour common benchmarks: GLUE, SuperGLUE, SQuAD, WMT
E3Key Metrics

Three automated metrics dominate generative AI evaluation, each targeting a different kind of output quality.

ROUGE
  • Measures overlap between generated and reference text — used for summarization quality
BLEU
  • Measures translation quality by comparing generated output to reference translations
BERTScore
  • Measures semantic similarity using contextual embeddings, catching paraphrases that ROUGE/BLEU miss
KeyCore metrics: ROUGE (summarization overlap), BLEU (translation quality), BERTScore (semantic similarity)
E4Additional Metrics

Two supporting metrics round out the evaluation toolkit.

KeyPerplexity: prediction confidence (lower = better)›F1 Score: balances precision and recall for classification-style outputs
F Deployment of Generative AI Applications F
F1Key Deployment Considerations

Deployment is where business value is actually realized — but only if four operational factors are planned for in advance.

Cost
  • Pay-per-use resource pricing that scales with actual traffic
Regions
  • Service availability differs by AWS region — latency and compliance both depend on this
Quotas
  • Built-in resource limits that must be checked against expected load
Security
  • Shared responsibility between AWS and the user, plus risks specific to external/third-party models
KeyFour deployment considerations: Cost, Regions, Quotas, Security›Security is shared responsibility — AWS secures the infrastructure, the user secures their configuration and data
F2Deployment Goals

A production-ready deployment is judged against four goals simultaneously — optimizing for only one at the expense of the others produces a fragile system.

Scalability
  • Handling growth in traffic and usage without redesign
Reliability
  • Consistent availability and correct behavior over time
Security
  • Protecting data, access, and model behavior from misuse
User Experience
  • Making the system fast, clear, and usable for its actual audience
KeyDeployment = where business value is realized›Four deployment goals: Scalability, Reliability, Security, User Experience
Module Summary
  • The Generative AI Application Lifecycle is iterative: Define → Select FM → Improve → Evaluate → Deploy, looping back as feedback arrives.
  • A strong use case definition — complete with actors, flows, and metrics — is the single biggest predictor of project success, not model choice.
  • Foundation Model selection is a trade-off across ten criteria (cost, latency, modality, size, responsibility, and more); model fit beats raw model power.
  • RAG, fine-tuning, prompting, and agents are complementary tools, not competing choices — most production systems combine RAG with a pre-trained model plus strong prompt engineering.
  • Evaluation must combine human judgment, benchmarks, and automated metrics (ROUGE, BLEU, BERTScore) before a model reaches deployment, where cost, regions, quotas, and shared-responsibility security determine real-world success.
